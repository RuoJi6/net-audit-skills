# net-auth-audit 评测样例

用于维护 description 触发边界和 Skill 行为。修改 `SKILL.md` 后，至少回看本文件中的样例。

## 应触发

| 用户输入 | 预期 | 理由 |
| --- | --- | --- |
| 帮我审计这个 .NET 项目的鉴权机制，看看有没有绕过或越权问题。 | 触发 | 明确要求 .NET 鉴权审计和绕过/越权分析 |
| 基于 net-route-mapper 的输出，分析每个接口是否有登录校验和权限校验。 | 触发 | 基于路由映射做接口鉴权状态分析 |
| 这个 ASP.NET MVC 项目只有 bin 和 web.config，先反编译 dll，再做鉴权审计。 | 触发 | 闭源 dll + web.config + 鉴权审计 |
| 检查 ASP.NET Core 项目里 `[Authorize]`、Policy、JWT 和资源归属校验是否完整。 | 触发 | ASP.NET Core 鉴权与资源级授权 |
| 帮我输出一份 .NET Web Forms 项目的鉴权审计报告。 | 触发 | Web Forms 鉴权审计报告 |

## 不应触发

| 用户输入 | 预期 | 理由 |
| --- | --- | --- |
| 帮我提取这个 ASP.NET 项目的所有路由和参数。 | 不触发 | 单纯路由提取，应使用 `net-route-mapper` |
| 解释一下 JWT 的结构和签名校验原理。 | 不触发 | 知识解释，不是项目审计 |
| 帮我给这个项目加一个登录注册功能。 | 不触发 | 开发任务，不是审计任务 |
| 审计这个接口有没有 SQL 注入。 | 不触发 | SQL 注入专项，不属于鉴权 Skill |
| 看看前端菜单权限为什么不显示。 | 不触发 | 前端权限显示问题，不是后端鉴权审计 |
| 检查 NuGet 依赖有没有 CVE。 | 不触发 | 通用依赖/CVE 审计不属于本 Skill |

## 边界例

| 用户输入 | 预期 | 理由 |
| --- | --- | --- |
| 这个接口没加 `[Authorize]`，是不是未授权？ | 触发但谨慎 | 需要检查全局 Filter、FallbackPolicy、基类、web.config，不能直接下结论 |
| 帮我看看 JWT 配置有没有问题，会不会导致接口越权。 | 触发 | 不是解释 JWT 原理，而是项目鉴权风险分析 |
| 检查前端菜单权限和后端接口鉴权是否一致。 | 触发 | 虽涉及前端，但目标是后端接口鉴权一致性 |
| 只列出所有匿名接口。 | 触发 | 属于鉴权状态分析的子任务，但仍要说明覆盖范围限制 |

## 失败案例

| 失败模式 | 错误行为 | 正确行为 |
| --- | --- | --- |
| 误报全局鉴权项目 | 看到 Action 没有 `[Authorize]` 就报未授权 | 先检查 `FallbackPolicy`、`DefaultPolicy`、`GlobalFilters`、基类、`web.config` |
| 漏报资源级越权 | 看到接口需要登录就标为安全 | 追踪 `userId`、`tenantId`、`deptId`、`companyId` 等资源归属校验 |
| 闭源审计跳过反编译 | 只有 `bin/*.dll` 就写泛泛建议 | 复用或生成共享反编译 manifest，再分析反编译产物 |
| 前端权限当后端授权 | 看到菜单/按钮有权限控制就认定接口安全 | 前端权限只作为线索，必须检查后端代码或配置证据 |
| Web Forms 漏页面基类 | 只看 `.aspx` 和 `web.config` | 追踪 `Inherits` 对应类、Page 基类、`OnInit`、`Page_Load` |
